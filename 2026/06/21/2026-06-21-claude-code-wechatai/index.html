从零搭建一个微信风格 AI 聊天应用（FastAPI + 原生 JS）

发表于2026-06-21|项目

项目地址：github.com/Abgnz1229/MychatBOT

开发方式：Claude Code（AI 编程助手）+ DeepSeek（LLM API）

技术栈：Python FastAPI · 原生 JavaScript · SSE 流式 · JSON 文件存储

缘起

市面上有很多 AI 聊天客户端，但要么太重，要么数据存在别人服务器上。我想做一个轻量、本地运行、数据完全掌控的聊天应用——微信风格的气泡界面，支持角色扮演，所有数据以 JSON 文件存储在本机，无须数据库。

整个开发过程使用 Claude Code 作为 AI 编程助手，API 调用走 DeepSeek。从第一行代码到完整可用的产品，全程在 Claude Code 的终端里完成——自然语言描述需求、AI 写代码、即时调试迭代。这篇文章本身也是 Claude Code 写的。

架构概览

设计原则：

  • 零依赖数据库 — 所有持久化用 JSON 文件，原子写入（write-to-tmp-then-rename）
  • 前后端分离但同仓库 — FastAPI 挂载静态文件，单命令启动
  • IIFE 模块模式 — 原生 JS，不引入构建工具，Chat / ChatState / API / Toast 四个全局模块

用户输入也有一个缓冲机制：用户可能在几秒内连发多条消息，前端用计时器收集后合并为一条发送，减少 API 调用。微信风格嘛，用户习惯了连发。

4. 人格预设系统

支持两种模式——简单设定（角色名 + 一段话）和角色卡（SillyTavern 兼容的 PNG/JSON）。统一序列化为 JSON 存到 personalities/ 目录：

  • type: "card" → 存完整 character_data（name, description, personality, scenario, first_mes, mes_example, system_prompt 等）
  • type: "simple" → 仅存 simple_name + simple_setting

加载人格时，card 类型直接重建 CharacterData 对象并赋值给 character._current_character，simple 类型写入 config。每次启动自动恢复上次使用的人格。

踩过的坑

  1. Windows GBK 编码 — json.load() 在中文 Windows 上默认用 GBK 打开文件，遇到中文就 UnicodeEncodeError。必须显式指定 encoding='utf-8'。

  2. SSE data: 跨 chunk 切割 — fetch 的 ReadableStream 不保证 chunk 边界对齐换行，一个 data: {"text": 可能被切成 da + ta: {"text":。必须用 lineBuffer 累计。

  3. API Key 泄露到 Git — 开发文档里写了真实 API Key 然后推上去了。事后补救：轮换 Key + gitignore 文档文件。教训：敏感信息一开始就要隔离。

  4. 前端 confirm() 误触 — confirm() 弹窗点击确认后没有乐观更新 DOM，用户以为没反应又点一次，导致两次 API 调用。修复：点击瞬间降低元素透明度 + finally 中移除 DOM。

  5. 空对话文件堆积 — 每次页面加载都创建新对话文件（含问候语），但用户可能没发消息就关掉了。修复：new_chat() 时不保存问候语到文件，list_chats() 过滤掉无用户消息的对话。

总结

这个项目证明了不用框架也能做出体验不错的 SPA。原生 JS IIFE 模块 + FastAPI + JSON 文件存储，整个应用不到 2000 行代码，功能却相当完整：多轮对话、流式输出、角色扮演、人格预设、对话管理。

适合的场景：个人使用、学习参考、二次开发。不适合的场景：多用户、高并发、生产部署——那应该上数据库和正经前端框架。

如果你也想做一个，Fork 下来改 config.json 就能跑。欢迎 PR。
